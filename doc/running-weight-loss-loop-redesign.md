# 跑步瘦身助手重设计

## 1. 新定位

当前产品不再只是“健康记录工具”，而应定位为：

> 跑步瘦身助手：通过每日跑步打卡、健康数据记录、目标差距反馈、预计达成时间和防摆烂提醒，帮助用户持续完成减肥循环。

它的核心不是“存数据”，而是让用户每天看到：

- 今天该不该跑
- 今天有没有记录
- 我离目标还差多少
- 我最近是不是真的在变瘦
- 按当前趋势还要多久
- 我不能继续摆烂的具体原因

## 2. 核心循环

新的产品循环是：

```text
设定目标
  -> 今日跑步/健康打卡
  -> 上传跑步和身体数据
  -> 首页看到差距缩小或风险变大
  -> 系统给出预计剩余天数和行动提醒
  -> 励志文案/对比图片强化坚持
  -> 第二天继续打卡
```

这个循环必须在首页第一屏就被看见，而不是藏在历史、趋势或设置里。

## 3. 首页重设计

首页是“减肥作战面板”，不是普通仪表盘。

### 3.1 首屏主区域：今日战况

首屏顶部直接展示：

- 今天是否跑步
- 今天是否记录健康数据
- 当前连续执行状态
- 今日主行动按钮

推荐文案：

- 已跑步：`今天已完成跑步，继续补齐身体数据。`
- 未跑步：`今天还没跑，先完成一次跑步打卡。`
- 未记录：`今天还没有身体数据，记录后才能判断是否真的在瘦。`

主按钮只保留一个：

- 未跑步时：`去上传跑步数据`
- 已跑步未记录健康时：`补充健康数据`
- 全部完成时：`查看今日变化`

### 3.2 第二层：目标卡片

首页第一屏的目标区固定展示两张卡片：

- 健康目标：目标体重、目标腰围、当前值、剩余差距、预计剩余天数。
- 运动目标：本周跑步次数目标、本周跑量目标、已完成次数、已跑距离、剩余差距。

不要再拆成“目标体重 / 目标腰围”两张独立卡片。体重和腰围属于同一类健康目标，放在一张健康目标卡里更利于用户理解；跑步次数和跑量属于同一类运动目标，放在一张运动目标卡里更利于行动。

健康目标卡显示：

- 目标值
- 当前真实值
- 剩余差距
- 上次变化
- 日均变化
- 预计剩余天数

示例：

```text
目标体重 70.0 公斤
当前 76.4 公斤
还差 6.4 公斤
较上次 -0.3 公斤
预计 48 天
```

目标腰围在同一卡片内展示：

```text
目标腰围 82.0 厘米
当前 89.5 厘米
还差 7.5 厘米
较上次 -0.4 厘米
预计 56 天
```

规则：

- 如果数据不足，不显示假预测，显示 `数据不足，至少需要 3 条有效记录`。
- 如果趋势反向，显示 `当前趋势没有靠近目标`。
- 如果达到目标，显示 `已达到目标，继续保持记录`。
- 未设置健康目标时使用蓝灰中性提示色，不使用大面积橙色或高饱和警告色。
- 健康目标已设置或达成时使用翠绿色系。
- 运动目标已设置或达成时使用运动蓝色系。
- 今日打卡未完成使用红色，完成使用翠绿色。

### 3.3 第三层：打卡和跑步驱动区

跑步是减肥动作源，不能只是普通数据。

首页下方继续展示今日打卡和跑步驱动信息：

- 本周跑步次数目标
- 本周已跑次数
- 本周跑量目标
- 本周已跑距离
- 还差几次 / 几公里

示例：

```text
本周跑步
已完成 2 / 4 次
已跑 9.8 / 20 公里
还差 2 次、10.2 公里
```

这里的按钮：

- `上传一次跑步`
- `查看跑步历史`

### 3.4 全局视觉动效

登录页、首次创建管理员页和登录后的应用页都需要体现轻量运动感：

- 登录页和首次创建管理员页使用全屏运动背景，并响应鼠标移动和点击。
- 登录后的所有主导航页共享同一套全视口运动背景和鼠标跟随效果，不能只在首页或中间内容容器内出现动效。
- 动效应像跑道、速度线、呼吸光感这类运动主题元素，避免抽象到看不出运动含义。
- 动效不能过于突兀，必须尊重 `prefers-reduced-motion`。

### 3.5 第四层：趋势反馈

趋势不再以复杂大图为主，而是以解释性反馈为主。

需要保留简洁趋势：

- 最近 7 天体重变化
- 最近 7 天腰围变化
- 最近 7 天跑步完成情况

但视觉重点应是文字解释：

```text
最近 7 天体重下降 0.8 公斤，腰围下降 1.2 厘米。
当前趋势有效，继续保持每周 4 次跑步。
```

而不是把用户扔进一堆曲线里自己理解。

### 3.5 第五层：防摆烂反馈

首页必须有一个“防摆烂提醒”区。

规则：

- 今天没跑：提醒跑步
- 连续 2 天没跑：更强提醒
- 有跑步但没健康数据：提醒补数据
- 趋势靠近目标：鼓励继续
- 趋势反向：提醒调整

文案要具体，不能羞辱。

示例：

```text
今天还没有跑步记录。你不是缺计划，是缺今天这一次行动。
先完成 20 分钟，数据会替你证明这件事没有白做。
```

```text
你最近 3 天没有跑步记录。目标不会因为计划存在而靠近。
今天先恢复一次，不需要完美，只需要重新开始。
```

## 4. 上传数据页重设计

上传页不是“表单页面”，而是“今日闭环完成页”。

### 4.1 页面顶部

展示今日完成状态：

```text
今日闭环
跑步：未完成
健康数据：已记录
目标反馈：待更新
```

### 4.2 健康数据

保持竖向录入，不使用巨大横向输入框。

字段：

- 体重
- 腰围
- 臀围
- 体脂率

录入后显示：

- 当前值
- 目标值
- 与目标差距
- 较上次变化

### 4.3 跑步数据

同样竖向录入。

字段：

- 公里数
- 运动时长
- 平均心率
- 平均步幅
- 步频
- 配速自动计算

提交后显示：

- 今日跑步次数
- 今日总公里数
- 本周进度变化

## 5. 目标页重设计

目标页应从“设置表单”升级为“目标计划页”。

### 5.1 健康目标

展示：

- 目标体重
- 目标腰围
- 目标体脂率
- 当前值
- 差距
- 数据是否足够预测

### 5.2 跑步目标

展示：

- 每周跑步次数
- 每周跑量
- 推荐每日最低行动

示例：

```text
为了完成每周 20 公里目标，剩余 4 天平均每天需要 2.6 公里。
```

## 6. 历史页重设计

历史页承担“证据”作用，不是主反馈。

它应该支持：

- 按周查看
- 按健康/跑步筛选
- 显示每周总结
- 显示编辑入口

历史页的核心问题是：

```text
我过去这一段到底做了什么？
```

不是：

```text
我现在该不该继续？
```

后者属于首页。

## 7. 设置页重设计

设置页保留工具属性，不抢主流程。

新增或强化：

- 提醒时间
- 防摆烂提醒强度
- 邮件提醒
- 对比图片配置

## 8. 对比图片提醒

这是新能力，需要谨慎设计。

### 8.1 图片类型

支持两类：

- 当前状态照片
- 目标激励照片

第一版可以只支持本地上传或填写图片 URL。

### 8.2 展示位置

不建议放在所有页面。

推荐位置：

- 首页防摆烂区
- 连续未跑步时的提醒邮件
- 设置页管理图片

### 8.3 文案原则

图片不能用于羞辱用户。

推荐表达：

```text
这不是为了否定现在的你，而是提醒你别忘了为什么开始。
```

## 9. 导航重设计

当前导航可调整为：

```text
首页
上传数据
历史证据
目标计划
提醒设置
系统设置
```

如果要保持简洁，也可以：

```text
首页
上传
历史
目标
设置
```

但页面内部文案必须围绕“跑步瘦身循环”。

## 10. 分阶段落地

### 阶段 1：首页闭环重构

目标：

- 首页第一屏改成“今日战况 + 健康目标 + 运动目标”
- 去掉泛泛的仪表盘优先级
- 让用户一打开就知道今天要做什么

### 阶段 2：上传页闭环重构

目标：

- 上传页顶部展示今日闭环状态
- 健康和跑步录入后直接显示差距变化
- 保持竖向、紧凑、低摩擦输入

### 阶段 3：预测与励志反馈

目标：

- 完善预计剩余天数
- 区分数据不足、趋势有效、趋势反向
- 加入基于真实数据的励志文案

### 阶段 4：防摆烂提醒

目标：

- 站内提醒变成“缺失行动提醒”
- 邮件提醒文案引用真实缺口
- 支持提醒强度

### 阶段 5：对比图片

目标：

- 支持配置对比图片
- 首页按条件展示
- 邮件提醒可带图片链接或提醒入口

## 11. BMAD 建议

这属于产品方向校正，不是普通 UI 微调。

BMAD 上建议：

- 使用 `bmad-correct-course` 记录这次方向变更。
- 使用 `bmad-ux` 重新生成核心页面体验规范。
- 当前如果继续直接实现，应按 `bmad-quick-dev` 分阶段落地，先做首页闭环重构。

下一步最合理的工程落点是：

```text
阶段 1：首页闭环重构
```

因为首页是循环启动器，只有首页先变成“行动 + 反馈 + 防放弃”的控制台，后续上传页、提醒页和图片能力才有清晰依附点。
